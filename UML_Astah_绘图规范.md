# UML 图绘制规范：Activity、Use Case、Component、Deployment、Package

本文面向使用 Astah 绘制 UML 图的场景，说明 Activity Diagram、Use Case Diagram、Component Diagram、Deployment Diagram、Package Diagram 的绘图标准、连线规则、布局习惯、命名规范和检查要点。

## 1. 通用绘图规范

### 1.1 基本原则

UML 图不是“把元素画出来”就结束，而是要清楚表达系统结构、行为或部署关系。绘图时应遵守以下原则：

- 一张图只表达一个明确主题，不要把所有内容塞进一张图。
- 图名要准确，例如“用户下单活动图”“会员系统用例图”“支付服务组件图”。
- 元素命名要使用业务或系统真实概念，避免“模块1”“功能A”“节点B”。
- 线条尽量少交叉，方向尽量一致。
- 同类元素使用同一层级、同一命名风格。
- 不要滥用注释，图本身应能表达主要含义。
- 图中只保留有助于理解的信息，细节过多时应拆分子图。

### 1.2 Astah 中的通用操作习惯

在 Astah 中绘制 UML 图时，通常先创建对应图类型，再从工具栏选择节点或关系线：

- 右键模型或包，选择 `Create Diagram` 创建图。
- 从左侧工具栏选择元素，例如 Activity、UseCase、Component、Node、Package 等。
- 使用关系线工具连接元素，例如 Association、Dependency、Include、Extend、Control Flow 等。
- 选中元素后可在属性区域修改名称、构造型、可见性、说明等。
- 可以使用对齐、自动布局、网格吸附等功能保持图面整洁。

### 1.3 命名规范

推荐命名方式如下：

| 元素类型 | 推荐命名 | 示例 |
|---|---|---|
| 活动 / 动作 | 动宾短语 | 提交订单、校验库存、生成支付单 |
| 用例 | 用户目标，动宾短语 | 注册账号、查询订单、申请退款 |
| 参与者 | 角色名或外部系统名 | 顾客、管理员、支付平台 |
| 组件 | 系统模块或服务名 | 订单服务、库存组件、支付接口 |
| 节点 | 运行环境或设备名 | Web 服务器、数据库服务器、用户终端 |
| 包 | 逻辑分组名 | 用户管理、订单域、基础设施层 |

不推荐：

- 名称过于技术化但缺少业务语义，例如“ProcessData”“Handler1”。
- 用例名称写成系统行为说明，例如“系统保存订单数据”，更推荐“提交订单”。
- 活动图动作写成名词，例如“订单校验”，更推荐“校验订单”。

### 1.4 线条规范

常见线条使用规则：

| 关系 | 线型 | 箭头 | 常见用途 |
|---|---|---|---|
| Association | 实线 | 可有方向 | 参与者与用例、类之间普通关系 |
| Dependency | 虚线 | 开口箭头 | 一个元素依赖另一个元素 |
| Generalization | 实线 | 空心三角箭头 | 继承、泛化 |
| Include | 虚线 | 箭头指向被包含用例 | 必然复用的公共用例 |
| Extend | 虚线 | 箭头指向被扩展用例 | 条件性扩展行为 |
| Control Flow | 实线 | 箭头指向下一个动作 | 活动流程 |
| Object Flow | 实线 | 箭头指向对象流向 | 活动中对象或数据流转 |
| Communication Path | 实线 | 通常无方向 | 部署节点之间通信连接 |

线条绘制讲究：

- 箭头方向必须表达“谁流向谁”“谁依赖谁”“谁调用谁”。
- 依赖关系箭头指向被依赖方。
- 泛化关系箭头指向父类、父用例、父组件或父节点。
- Include 箭头从基础用例指向被包含用例。
- Extend 箭头从扩展用例指向被扩展用例。
- 活动图控制流箭头按业务流程前进方向连接。
- 尽量避免斜线和交叉线，优先使用水平或垂直折线。

### 1.5 版面规范

推荐布局：

- 从左到右表示调用、依赖或交互方向。
- 从上到下表示时间顺序或流程推进。
- 参与者放在用例图边界外。
- 系统内部元素放在系统边界内。
- 高层元素放上方，低层或被依赖元素放下方。
- 物理部署图中用户终端、服务器、数据库可按网络拓扑层次摆放。

避免：

- 元素大小不一致且无理由。
- 线条大量交叉。
- 箭头方向混乱。
- 图中同时表达业务流程、模块结构、部署结构，导致主题不清。

## 2. Activity Diagram 活动图

### 2.1 图的用途

Activity Diagram 用于描述业务流程、系统流程、操作步骤、条件分支、并发处理和对象流转。它关注“过程如何发生”，适合描述：

- 用户下单流程。
- 审批流程。
- 登录认证流程。
- 支付退款流程。
- 批处理任务流程。
- 多角色协作流程。

活动图不适合表达系统静态结构，例如模块依赖、部署节点、类关系。

### 2.2 Astah 中常用元素

| 元素 | 含义 | Astah 中的使用 |
|---|---|---|
| Initial Node | 起点 | 一个流程通常只有一个起点 |
| Activity Final Node | 活动终点 | 表示整个活动结束 |
| Flow Final Node | 流终点 | 表示某一条流程分支结束 |
| Action | 动作 | 表示一个具体步骤 |
| Decision Node | 决策 | 根据条件选择分支 |
| Merge Node | 合并 | 多个可选分支重新汇合 |
| Fork Node | 分叉 | 一个流程拆成并发流程 |
| Join Node | 汇合 | 多个并发流程同步后继续 |
| Swimlane / Partition | 泳道 | 按角色、系统或部门划分职责 |
| Object Node | 对象节点 | 表示流程中的数据或对象 |
| Control Flow | 控制流 | 表示动作执行顺序 |
| Object Flow | 对象流 | 表示对象、数据、文档的传递 |

### 2.3 绘制步骤

1. 明确流程边界：流程从哪里开始，到哪里结束。
2. 列出主要动作：每一步用动宾短语表达。
3. 判断是否需要泳道：如果涉及多个角色、系统或部门，应使用泳道。
4. 放置起点和终点。
5. 按执行顺序放置 Action。
6. 在条件判断处使用 Decision Node。
7. 在分支汇合处使用 Merge Node。
8. 在并发处理处使用 Fork / Join。
9. 用 Control Flow 连接动作。
10. 给分支线标注条件，例如 `[库存充足]`、`[库存不足]`。
11. 检查是否每个分支都有出口。

### 2.4 连线规则

#### 控制流 Control Flow

控制流表示动作执行顺序，使用实线箭头。

规范：

- 起点只能流向第一个动作或决策节点。
- 一个普通动作通常只有一个进入流和一个流出流。
- 箭头方向表示执行顺序。
- 不要用双向箭头。
- 不要让一个动作无原因地直接连向多个动作，分支应使用 Decision Node 或 Fork Node。

示例：

```text
开始 -> 填写订单 -> 提交订单 -> 校验库存 -> 结束
```

#### 决策与合并

Decision Node 用菱形表示条件分支。

规范：

- 决策节点通常有一个进入流，多个流出流。
- 每条流出线都应标注守卫条件。
- 守卫条件写在方括号中，例如 `[通过]`、`[失败]`。
- 条件应互斥且完整。
- 分支重新汇合时应使用 Merge Node。

示例：

```text
校验库存 -> 库存判断
库存判断 -> [库存充足] 创建订单
库存判断 -> [库存不足] 提示缺货
```

不要把条件直接写在动作名称里，例如“如果库存充足则创建订单”。应该使用 Decision Node 和条件线。

#### 并发 Fork / Join

Fork 和 Join 通常用粗横线或竖线表示。

规范：

- Fork：一个进入流，多个并发流出流。
- Join：多个进入流，一个流出流。
- Fork 和 Join 要成对使用，除非业务上明确允许异步结束。
- 并发分支之间不要画控制流，除非存在明确依赖。

示例：

```text
订单已创建 -> Fork
Fork -> 扣减库存
Fork -> 发送通知
Fork -> 生成发票
Join -> 更新订单状态
```

#### 对象流 Object Flow

Object Flow 表示数据、文档或对象在活动之间传递。

规范：

- 对象节点使用名词，例如“订单”“支付单”“审批结果”。
- 动作使用动词，例如“生成订单”“确认支付”。
- 对象流不应替代控制流。
- 对象流适合补充说明数据如何产生、传递和消费。

### 2.5 泳道规范

泳道用于划分职责。常见划分方式：

- 按角色：顾客、客服、管理员。
- 按系统：前端、订单服务、支付系统、库存系统。
- 按部门：销售部、财务部、仓储部。

规范：

- 泳道名称应是责任主体，不是动作。
- 一个动作放在真正负责执行它的泳道内。
- 跨泳道连线表示职责交接或系统调用。
- 泳道不宜过多，超过 5 个时建议拆图。

### 2.6 常见错误

- 没有起点或终点。
- 多个分支没有条件说明。
- 用普通箭头直接分出多个流程，却没有 Decision 或 Fork。
- 把系统结构图画成活动图。
- 动作名称全部是名词，缺少流程含义。
- 泳道划分混乱，同一角色动作分散在多个泳道。

### 2.7 检查清单

- 是否有明确的开始和结束？
- 每个动作是否是动宾短语？
- 条件分支是否标注了守卫条件？
- 并发流程是否使用 Fork / Join？
- 跨角色流程是否使用泳道？
- 是否存在无来源或无去向的线？
- 是否有线条交叉过多的问题？

## 3. Use Case Diagram 用例图

### 3.1 图的用途

Use Case Diagram 用于描述系统对外提供的功能，以及外部参与者如何与系统交互。它关注“谁使用系统做什么”，适合用于需求分析阶段。

用例图表达：

- 系统边界。
- 外部参与者。
- 系统提供的用户目标。
- 用例之间的包含、扩展、泛化关系。

用例图不表达：

- 功能执行顺序。
- 数据表结构。
- 页面跳转细节。
- 内部模块调用。

### 3.2 Astah 中常用元素

| 元素 | 含义 | 使用规范 |
|---|---|---|
| Actor | 参与者 | 系统外部的人、组织或系统 |
| Use Case | 用例 | 参与者希望达成的目标 |
| System Boundary | 系统边界 | 表示当前系统范围 |
| Association | 关联 | 参与者参与某个用例 |
| Include | 包含 | 基础用例必然执行公共用例 |
| Extend | 扩展 | 特定条件下扩展基础用例 |
| Generalization | 泛化 | 角色或用例的继承关系 |

### 3.3 绘制步骤

1. 确定系统边界，画出系统框并命名。
2. 找出系统外部参与者。
3. 找出参与者的目标，形成用例。
4. 将用例放入系统边界内。
5. 将参与者放在系统边界外。
6. 用 Association 连接参与者和用例。
7. 提取公共必选行为，使用 Include。
8. 提取条件性或可选行为，使用 Extend。
9. 对相似参与者或用例使用 Generalization。

### 3.4 参与者 Actor 规范

参与者是系统外部与系统交互的角色，不一定是人，也可以是外部系统。

示例：

- 顾客。
- 管理员。
- 支付平台。
- 短信服务。
- 第三方物流系统。

规范：

- 参与者放在系统边界外。
- 参与者名称使用角色名，不使用具体人名。
- 不要把系统内部模块画成 Actor。
- 如果某外部系统通过接口调用当前系统，可以作为 Actor。

### 3.5 用例 Use Case 规范

用例表示参与者通过系统完成的目标。

规范：

- 用例名称使用动宾短语。
- 用例应体现用户可感知的目标。
- 用例粒度不要太细，不要把每个按钮都画成一个用例。
- 用例也不要太大，例如“管理系统”通常不是好用例。

推荐：

- 注册账号。
- 登录系统。
- 提交订单。
- 支付订单。
- 申请退款。
- 审核退款申请。

不推荐：

- 点击提交按钮。
- 输入用户名。
- 保存数据。
- 用户模块。
- 订单功能。

### 3.6 系统边界规范

系统边界用于说明当前图描述的是哪个系统。

规范：

- 系统边界名称应是系统或子系统名称。
- 用例放在边界内。
- 参与者放在边界外。
- 如果描述多个系统之间关系，应谨慎使用边界，避免把外部系统误放入当前系统。

### 3.7 连线规则

#### Association 关联

Association 表示参与者参与某个用例。

规范：

- 使用实线连接 Actor 和 Use Case。
- 通常不需要箭头。
- 只连接参与者和用例，不要随意连接用例和用例。
- 一个参与者可以关联多个用例。
- 一个用例可以关联多个参与者。

#### Include 包含关系

Include 表示基础用例每次执行时，都必须执行被包含用例。

规范：

- 使用虚线箭头。
- 标注 `<<include>>`。
- 箭头从基础用例指向被包含用例。
- 被包含用例通常是多个用例共同复用的必选步骤。

示例：

```text
提交订单 --<<include>>--> 校验用户身份
支付订单 --<<include>>--> 校验用户身份
```

适合 Include 的场景：

- 登录校验。
- 权限校验。
- 记录操作日志。
- 计算订单金额。

不适合 Include 的场景：

- 某些情况下才发生的行为。
- 用户可选操作。
- 异常处理分支。

#### Extend 扩展关系

Extend 表示某个用例在特定条件下扩展另一个用例。

规范：

- 使用虚线箭头。
- 标注 `<<extend>>`。
- 箭头从扩展用例指向被扩展用例。
- 扩展用例通常是可选、异常或条件触发行为。

示例：

```text
使用优惠券 --<<extend>>--> 提交订单
申请发票 --<<extend>>--> 提交订单
```

适合 Extend 的场景：

- 可选功能。
- 异常流程。
- 条件触发功能。
- 对基础用例的补充行为。

#### Generalization 泛化关系

Generalization 表示参与者或用例之间的继承关系。

规范：

- 使用实线空心三角箭头。
- 箭头指向父参与者或父用例。
- 子参与者继承父参与者关联的用例。

示例：

```text
普通管理员 --|> 管理员
超级管理员 --|> 管理员
```

### 3.8 Include 与 Extend 的区别

| 对比项 | Include | Extend |
|---|---|---|
| 是否必然发生 | 必然发生 | 条件发生 |
| 箭头方向 | 基础用例 -> 被包含用例 | 扩展用例 -> 被扩展用例 |
| 目的 | 抽取公共行为 | 表达可选或异常行为 |
| 常见例子 | 身份校验、权限检查 | 使用优惠券、申请发票 |

### 3.9 常见错误

- 把系统内部模块画成参与者。
- 用例粒度太细，变成页面操作清单。
- 用例名称写成名词。
- Include 和 Extend 箭头方向画反。
- 系统边界外放置了本系统用例。
- 用例图里表达复杂流程顺序。

### 3.10 检查清单

- 系统边界是否清楚？
- 参与者是否都在系统外部？
- 用例是否都是用户目标？
- 用例名称是否是动宾短语？
- Include 是否表示必选公共行为？
- Extend 是否表示条件或可选行为？
- 箭头方向是否正确？

## 4. Component Diagram 组件图

### 4.1 图的用途

Component Diagram 用于描述系统的软件组件、接口、依赖关系和模块组织。它关注“系统由哪些可替换、可部署、可复用的组件组成，以及它们如何依赖”。

适合表达：

- 服务之间依赖。
- 模块之间接口关系。
- 分层架构。
- 子系统划分。
- 第三方组件依赖。
- 微服务架构。

不适合表达：

- 业务流程顺序。
- 物理服务器部署。
- 用户和系统功能关系。

### 4.2 Astah 中常用元素

| 元素 | 含义 | 使用规范 |
|---|---|---|
| Component | 组件 | 可替换的软件单元 |
| Interface | 接口 | 组件提供或需要的服务 |
| Provided Interface | 提供接口 | 通常用棒棒糖符号表示 |
| Required Interface | 需要接口 | 通常用插口符号表示 |
| Port | 端口 | 组件对外交互点 |
| Dependency | 依赖 | 一个组件依赖另一个组件 |
| Realization | 实现 | 组件实现接口 |
| Package | 包 | 对组件进行分组 |
| Artifact | 制品 | jar、war、dll、镜像等可部署文件 |

### 4.3 绘制步骤

1. 确定组件粒度：服务、模块、库、子系统或可部署单元。
2. 画出主要 Component。
3. 标出组件提供的接口。
4. 标出组件依赖的接口或组件。
5. 用 Dependency 或接口连接表达关系。
6. 用 Package 对组件进行分层或领域分组。
7. 必要时补充 Artifact，说明组件对应的部署制品。

### 4.4 组件命名规范

组件名称应体现职责或部署单元。

推荐：

- 用户服务。
- 订单服务。
- 支付组件。
- 库存管理模块。
- 报表生成组件。
- 消息通知服务。

不推荐：

- Service1。
- DataProcessor。
- Common。
- Utils。
- 模块A。

如果是技术组件，可以使用明确技术名：

- Redis 缓存组件。
- MySQL 数据访问组件。
- Kafka 消息组件。

### 4.5 接口规范

接口表示组件对外提供或需要的能力。

常见画法：

- Provided Interface：组件提供给外部使用的接口，通常画成圆形棒棒糖。
- Required Interface：组件需要外部提供的接口，通常画成半圆插口。
- Interface Classifier：也可以画成单独接口元素，并用 Realization 或 Dependency 连接。

规范：

- 接口命名通常使用能力名或服务名，例如 `IOrderService`、`Payment API`。
- 组件对外暴露的能力应通过接口表达，不建议直接画大量组件互相依赖。
- 如果图偏架构说明，可以使用简化依赖线，不必画所有端口。
- 如果图偏接口设计，应清楚标出 provided / required interface。

### 4.6 连线规则

#### Dependency 依赖

Dependency 表示一个组件依赖另一个组件。

规范：

- 使用虚线开口箭头。
- 箭头从依赖方指向被依赖方。
- 如果订单服务调用库存服务，箭头应从订单服务指向库存服务。

示例：

```text
订单服务 ..> 库存服务
订单服务 ..> 支付服务
```

#### Interface Realization 接口实现

组件实现接口时，可使用 Realization。

规范：

- 使用虚线空心三角箭头。
- 箭头指向接口。
- 表示组件实现该接口。

示例：

```text
订单服务 ..|> IOrderService
```

#### Provided / Required Interface 连接

组件通过接口交互时：

- 提供方使用 Provided Interface。
- 使用方使用 Required Interface。
- 两者可以通过 Assembly Connector 连接。

规范：

- 提供接口放在提供组件边缘。
- 需要接口放在使用组件边缘。
- 接口名称要清楚标注。
- 不要把所有接口都画出来，只画对当前图主题有意义的接口。

### 4.7 分层组件图规范

如果用组件图表达分层架构，可按层次从上到下布局：

```text
表现层
应用层
领域层
基础设施层
外部系统
```

规范：

- 上层可以依赖下层。
- 下层不应反向依赖上层。
- 跨层依赖应谨慎标注。
- 公共组件放在独立区域，不要混在业务组件中。

### 4.8 常见错误

- 把类图当组件图画，组件粒度太小。
- 把部署服务器画进组件图，混淆组件图和部署图。
- 依赖箭头方向画反。
- 所有组件互相连接，形成网状图但无法读懂。
- 没有接口，只有一堆模块方框。
- `Common`、`Utils` 等泛化组件过多，职责不清。

### 4.9 检查清单

- 组件粒度是否一致？
- 组件名称是否表达职责？
- 依赖箭头是否从使用方指向被使用方？
- 重要接口是否明确？
- 是否混入了物理部署细节？
- 分层关系是否清晰？
- 是否存在不合理的循环依赖？

## 5. Deployment Diagram 部署图

### 5.1 图的用途

Deployment Diagram 用于描述软件系统运行时的物理或虚拟部署结构。它关注“软件部署在哪里、节点之间如何通信、制品如何分布”。

适合表达：

- 服务器部署结构。
- 容器、虚拟机、云服务部署。
- 客户端、网关、应用服务、数据库之间的连接。
- 软件制品部署到哪些节点。
- 运行环境和通信协议。

不适合表达：

- 业务流程。
- 用户需求功能。
- 组件内部代码结构。

### 5.2 Astah 中常用元素

| 元素 | 含义 | 使用规范 |
|---|---|---|
| Node | 运行节点 | 物理机器、虚拟机、容器、设备 |
| Device | 硬件设备 | 手机、PC、服务器、网关 |
| Execution Environment | 执行环境 | JVM、Docker、Tomcat、Kubernetes Pod |
| Artifact | 部署制品 | jar、war、镜像、配置文件 |
| Component | 软件组件 | 可部署的软件单元 |
| Communication Path | 通信路径 | 节点之间的网络连接 |
| Dependency | 依赖 | 制品或节点之间依赖 |

### 5.3 绘制步骤

1. 明确部署范围：开发环境、测试环境、生产环境还是云架构。
2. 画出物理或虚拟节点。
3. 在节点内放置执行环境，例如 Docker、JVM、Tomcat。
4. 在执行环境内放置 Artifact 或 Component。
5. 用 Communication Path 连接节点。
6. 在线上标注协议、端口或网络类型。
7. 如有集群、负载均衡、主从数据库，应明确标注。

### 5.4 节点规范

节点表示运行时资源，可以是硬件，也可以是虚拟运行环境。

常见节点：

- 用户终端。
- Web 服务器。
- 应用服务器。
- 数据库服务器。
- 缓存服务器。
- 消息队列服务器。
- 文件存储。
- Kubernetes 集群。
- Docker 容器。

规范：

- 节点名称应表达部署环境或资源角色。
- 可以使用构造型标注，例如 `<<device>>`、`<<executionEnvironment>>`。
- 物理节点可以包含执行环境。
- 执行环境可以包含部署制品。

示例层次：

```text
<<device>> 应用服务器
  <<executionEnvironment>> Docker
    order-service.jar
```

### 5.5 Artifact 制品规范

Artifact 表示实际部署的文件或包。

示例：

- `order-service.jar`
- `payment-service.war`
- `frontend.zip`
- `nginx.conf`
- `mysql:8.0 image`
- `order-service:v1.3.2 image`

规范：

- Artifact 放在它实际部署的节点或执行环境内。
- Artifact 名称可以包含版本，但不必在高层图中频繁写版本号。
- 如果同一个制品部署多份，应通过节点复制或标注副本数量表达。

### 5.6 连线规则

#### Communication Path 通信路径

Communication Path 表示节点之间可以通信。

规范：

- 使用实线连接节点。
- 可在线上标注协议和端口。
- 方向可选；如果需要表达调用方向，可加箭头或使用依赖关系辅助说明。

示例：

```text
用户浏览器 -- HTTPS/443 -- Nginx
Nginx -- HTTP/8080 -- 应用服务
应用服务 -- JDBC/3306 -- MySQL
应用服务 -- Redis/6379 -- Redis
```

#### Dependency 依赖

部署图中 Dependency 可用于表达制品依赖或部署依赖。

规范：

- 使用虚线箭头。
- 箭头指向被依赖方。
- 不要用 Dependency 替代所有网络连接。

### 5.7 云部署和容器部署画法

云环境中可以把云资源画成节点或包围结构：

```text
<<cloud>> 阿里云 VPC
  <<node>> 负载均衡 SLB
  <<node>> Kubernetes 集群
    <<executionEnvironment>> Pod: order-service
      order-service image
  <<node>> RDS MySQL
  <<node>> Redis
```

规范：

- 外层表示网络或云环境，例如 VPC、Region、Availability Zone。
- 中层表示集群、节点、容器运行环境。
- 内层表示 Artifact 或 Component。
- 通信线标注协议、端口或服务名。
- 对高可用结构标注副本数，例如 `replicas=3`。

### 5.8 常见错误

- 把业务流程画进部署图。
- 只画软件模块，不画运行节点。
- 没有标注通信协议，无法看出系统如何连接。
- 把数据库画成普通组件，而不是部署节点或外部服务。
- 节点嵌套关系混乱。
- 生产、测试、开发环境混在一张图中。

### 5.9 检查清单

- 是否明确了部署环境？
- 节点是否代表真实运行资源？
- Artifact 是否放在正确节点内？
- 节点之间是否标注通信协议或端口？
- 是否表达了负载均衡、集群、副本等关键信息？
- 是否混入了过多业务流程或代码结构？

## 6. Package Diagram 包图

### 6.1 图的用途

Package Diagram 用于描述模型、代码、模块或业务领域的分组关系和依赖关系。它关注“系统如何分组，以及分组之间如何依赖”。

适合表达：

- 代码包依赖。
- 业务域划分。
- 分层架构。
- 子系统组织。
- 模型元素归属。

不适合表达：

- 具体业务流程。
- 物理部署。
- 详细接口调用。

### 6.2 Astah 中常用元素

| 元素 | 含义 | 使用规范 |
|---|---|---|
| Package | 包 | 一组相关模型元素 |
| Dependency | 依赖 | 一个包依赖另一个包 |
| Package Import | 包导入 | 导入另一个包的公开成员 |
| Package Merge | 包合并 | 合并另一个包的内容 |
| Containment | 包含 | 包中包含子包或元素 |

### 6.3 绘制步骤

1. 确定分组维度：按层、按领域、按子系统或按代码包。
2. 画出主要 Package。
3. 将子包或相关元素放入对应 Package。
4. 用 Dependency 表示包之间依赖。
5. 如果表达模型导入，使用 Package Import。
6. 如果表达模型扩展合并，使用 Package Merge。
7. 检查是否存在循环依赖。

### 6.4 包命名规范

包名应表示清晰的分组概念。

推荐：

- 用户管理。
- 订单管理。
- 支付域。
- 库存域。
- 应用层。
- 领域层。
- 基础设施层。

代码包可使用实际包名：

```text
com.example.order
com.example.payment
com.example.infrastructure
```

不推荐：

- 包1。
- 公共。
- 其他。
- 临时。
- 新建包。

### 6.5 包依赖规则

Dependency 表示一个包中的元素使用另一个包中的元素。

规范：

- 使用虚线箭头。
- 箭头从依赖包指向被依赖包。
- 包图重点是宏观依赖，不要画太多细节线。
- 如果包之间有大量双向依赖，应考虑重新划分边界。

示例：

```text
订单管理 ..> 用户管理
订单管理 ..> 支付管理
订单管理 ..> 库存管理
应用层 ..> 领域层
基础设施层 ..> 领域层
```

### 6.6 分层包图规范

分层包图通常按从上到下布局：

```text
接口层 / 表现层
应用层
领域层
基础设施层
```

规范：

- 上层可以依赖下层。
- 下层不应依赖上层。
- 同层包之间依赖应尽量少。
- 基础设施层可实现领域层定义的接口，但要注意依赖方向。

在领域驱动设计中，推荐让领域层保持稳定，不依赖具体技术实现。基础设施层可以依赖领域层并实现其接口。

### 6.7 Package Import 与 Package Merge

#### Package Import

Package Import 表示当前包导入另一个包的公共成员。

规范：

- 使用虚线箭头。
- 标注 `<<import>>`。
- 箭头指向被导入包。
- 适合表达模型层面的可见性引入。

#### Package Merge

Package Merge 表示一个包合并另一个包的内容，常用于模型扩展。

规范：

- 使用虚线箭头。
- 标注 `<<merge>>`。
- 箭头指向被合并包。
- 普通业务建模较少使用，除非确实需要表达模型扩展。

### 6.8 常见错误

- 包图画成组件图，过度关注接口和实现。
- 包粒度不一致，有的表示领域，有的表示类文件夹。
- 依赖箭头方向画反。
- 包之间循环依赖严重但没有说明。
- 包名使用“公共”“工具”“其他”等模糊概念。
- 在包图中画大量类之间关系，导致图面混乱。

### 6.9 检查清单

- 包的划分维度是否一致？
- 包名是否清楚表达职责？
- 依赖箭头是否从使用方指向被使用方？
- 是否存在循环依赖？
- 分层关系是否合理？
- 图中是否包含过多类级别细节？

## 7. 五类图的区别与选择

| 图类型 | 关注点 | 主要元素 | 适用问题 |
|---|---|---|---|
| Activity Diagram | 流程如何执行 | 动作、决策、分支、泳道 | 业务流程、系统流程 |
| Use Case Diagram | 谁使用系统做什么 | 参与者、用例、系统边界 | 需求范围、用户目标 |
| Component Diagram | 软件组件如何组织和依赖 | 组件、接口、依赖 | 架构设计、模块关系 |
| Deployment Diagram | 软件部署在哪里 | 节点、执行环境、制品、通信路径 | 物理部署、运行环境 |
| Package Diagram | 模型或代码如何分组 | 包、依赖、导入、合并 | 分层、领域、代码包组织 |

快速判断：

- 想表达“流程步骤”时，画 Activity。
- 想表达“用户能做什么”时，画 Use Case。
- 想表达“系统由哪些软件组件组成”时，画 Component。
- 想表达“软件部署在哪些机器或环境上”时，画 Deployment。
- 想表达“代码、模型或领域如何分组”时，画 Package。

## 8. Astah 绘图落地建议

### 8.1 建议的建模顺序

对于一个完整系统，可以按以下顺序绘制：

1. Use Case Diagram：确定系统边界和功能范围。
2. Activity Diagram：展开关键业务流程。
3. Package Diagram：划分业务域、层次或代码包。
4. Component Diagram：设计软件组件和接口依赖。
5. Deployment Diagram：说明组件和制品如何部署到运行环境。

这个顺序符合从需求到设计再到部署的建模过程。

### 8.2 Astah 中的模型管理规范

建议在 Astah 的模型树中按图类型或系统模块管理：

```text
项目模型
  需求视图
    用例图
  业务流程视图
    活动图
  架构设计视图
    包图
    组件图
  部署视图
    部署图
```

也可以按业务域管理：

```text
订单域
  订单用例图
  提交订单活动图
  订单组件图
支付域
  支付用例图
  支付流程活动图
  支付组件图
```

### 8.3 图面美观规范

- 同层元素横向或纵向对齐。
- 同类元素大小保持一致。
- 线条尽量使用直角折线。
- 避免线条穿过元素。
- 避免关系线文字压在节点上。
- 重要元素放在视觉中心。
- 外部系统或外部参与者放在图的边缘。
- 图过大时拆成总览图和细节图。

### 8.4 说明文字规范

可以使用 Note 补充说明，但应控制数量。

适合写 Note 的内容：

- 业务规则。
- 条件约束。
- 部署假设。
- 省略说明。
- 关键设计决策。

不适合写 Note 的内容：

- 把所有流程步骤写成大段文字。
- 重复图中已经表达清楚的信息。
- 放置和图主题无关的背景介绍。

### 8.5 版本和交付规范

交付 UML 图时建议包含：

- 图名称。
- 图类型。
- 适用系统或模块。
- 作者和日期。
- 版本号。
- 简短说明。

示例：

```text
图名：用户下单活动图
图类型：Activity Diagram
系统：电商交易系统
版本：v1.0
日期：2026-05-17
说明：描述顾客从提交订单到支付完成的主流程和异常分支。
```

## 9. 常用关系箭头速查

| 场景 | 使用关系 | 线型 | 箭头方向 |
|---|---|---|---|
| 活动执行顺序 | Control Flow | 实线 | 指向下一个动作 |
| 活动数据传递 | Object Flow | 实线 | 指向接收对象或动作 |
| 参与者使用用例 | Association | 实线 | 通常无箭头 |
| 必选公共用例 | Include | 虚线 | 基础用例指向被包含用例 |
| 可选扩展用例 | Extend | 虚线 | 扩展用例指向被扩展用例 |
| 继承 / 泛化 | Generalization | 实线空心三角 | 子元素指向父元素 |
| 组件依赖 | Dependency | 虚线 | 使用方指向被使用方 |
| 组件实现接口 | Realization | 虚线空心三角 | 实现方指向接口 |
| 包依赖 | Dependency | 虚线 | 依赖包指向被依赖包 |
| 节点通信 | Communication Path | 实线 | 可无方向，必要时标注调用方向 |

## 10. 最终审图清单

在 Astah 中完成图后，建议逐项检查：

- 图名是否清楚说明主题？
- 图类型是否选对？
- 元素命名是否统一？
- 箭头方向是否符合 UML 语义？
- 线条是否过度交叉？
- 是否有孤立元素？
- 是否有未命名或命名含糊的元素？
- 是否混入其他图类型才应该表达的信息？
- 图是否过大，是否需要拆分？
- 是否能让没有参与绘图的人读懂主要意思？

## 11. 简短示例：电商系统五类图如何分工

以电商下单为例：

- Use Case Diagram：画“顾客”可以“浏览商品、提交订单、支付订单、申请退款”，“管理员”可以“审核退款”。
- Activity Diagram：画顾客提交订单后的流程，包括校验库存、创建订单、发起支付、支付成功或失败分支。
- Package Diagram：画用户域、商品域、订单域、支付域、库存域之间的依赖。
- Component Diagram：画订单服务、库存服务、支付服务、消息通知服务、数据库访问组件之间的依赖和接口。
- Deployment Diagram：画浏览器、Nginx、应用服务器、订单服务容器、MySQL、Redis、消息队列之间的部署和通信。

这样五类图各自表达不同层面，组合起来才能完整说明系统。
