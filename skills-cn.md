# Codex Skills 中文功能目录

这份目录把当前可用的 skills 按功能场景重新排序，并把英文说明汉化成便于检索的中文速查。  
使用时可以直接点名 skill 名称，例如“用 `prd-development` 帮我写 PRD”。

## 1. Codex 能力管理与创作

| Skill | 中文说明 | 适合什么时候用 |
| --- | --- | --- |
| `skill-creator` | 创建或更新 Codex skill 的通用指南。 | 想把某种工作流沉淀成一个新 skill。 |
| `skill-authoring-workflow` | 把原始 PM 内容整理成合规、可发布的 skill。 | 已经有一堆方法论材料，要做成标准 skill。 |
| `skill-installer` | 从 curated 列表或 GitHub 仓库安装 Codex skills。 | 想列出、安装、更新外部 skill。 |
| `plugin-creator` | 创建 Codex 插件目录、插件元数据和基础结构。 | 想做一个本地 Codex plugin。 |
| `openai-docs` | 查询 OpenAI 官方 API / 产品文档并给出引用。 | 需要最新 OpenAI API、模型、迁移或提示词升级信息。 |
| `full-output-enforcement` | 强制完整输出，避免省略、占位符和中途截断。 | 需要完整代码、完整文档、完整清单。 |

## 2. 设计、前端与视觉生成

| Skill | 中文说明 | 适合什么时候用 |
| --- | --- | --- |
| `design-taste-frontend` | 高级 UI/UX 与前端设计工程规范，避免普通 AI 味界面。 | 要做或改一个认真可用、视觉成熟的网页或应用。 |
| `redesign-existing-projects` | 升级已有网站/应用的视觉质量，同时保持功能不坏。 | 已有项目看起来普通，需要高级感重设计。 |
| `high-end-visual-design` | 高端 agency 风格设计规范：字体、间距、阴影、卡片、动画等。 | 想让网站/界面显得更贵、更成熟。 |
| `minimalist-ui` | 干净的编辑型极简界面，暖调单色、排版对比、扁平 bento。 | 需要安静、克制、内容优先的界面。 |
| `industrial-brutalist-ui` | 工业粗粝风 UI：瑞士网格、终端感、机械感、强对比。 | 数据看板、作品集、编辑页面需要“解密档案/机械系统”气质。 |
| `gpt-taste` | 高级 GSAP 动效和网页叙事设计，强调滚动触发、AIDA、编辑排版。 | 想做强动效、高冲击力的营销页或作品页。 |
| `stitch-design-taste` | 为 Google Stitch 生成高质量 DESIGN.md 设计系统。 | 要让 Stitch 按高级审美稳定产出界面。 |
| `image-to-code` | 先生成设计图，再深度分析并实现网页，尽量贴近视觉稿。 | 视觉很重要，想从设计图到网页实现。 |
| `imagegen` | 生成或编辑 raster 图片：照片、插画、纹理、sprite、mockup 等。 | 需要 AI 生成 bitmap 图片或改图。 |
| `imagegen-frontend-web` | 为网站每个 section 单独生成高质量横向设计参考图。 | 做 landing page 或网页视觉方向探索。 |
| `imagegen-frontend-mobile` | 生成高质量移动 App 界面概念图和多屏流程。 | 做 iOS/Android/mobile 产品视觉稿。 |
| `brandkit` | 生成高级品牌手册、logo 系统、identity board、视觉世界。 | 做品牌视觉、logo 方向、品牌 guideline。 |

## 4. 产品发现、用户研究与问题定义

| Skill | 中文说明 | 适合什么时候用 |
| --- | --- | --- |
| `discovery-process` | 从问题假设到验证方案，跑完整产品发现流程。 | 团队需要系统做 discovery。 |
| `discovery-interview-prep` | 规划客户访谈目标、对象、约束和方法。 | 准备问题验证、流失研究、新产品访谈。 |
| `jobs-to-be-done` | 挖掘用户 Jobs、痛点和收益。 | 做需求澄清、定位、发现未满足需求。 |
| `customer-journey-map` | 绘制客户旅程：阶段、触点、行为、情绪、指标。 | 诊断体验断点或对齐完整用户流程。 |
| `customer-journey-mapping-workshop` | 交互式客户旅程工作坊。 | 想一步步产出 persona 场景下的旅程图。 |
| `proto-persona` | 基于现有研究、市场信号和团队认知创建 proto-persona。 | 缺少完整研究，但需要临时用户画像。 |
| `problem-statement` | 写用户中心的问题陈述：谁被卡住、想做什么、为什么重要。 | 立项、PRD、优先级讨论前澄清问题。 |
| `problem-framing-canvas` | 使用 MITRE Problem Framing Canvas 梳理问题。 | 团队过早跳方案，需要先框定问题。 |
| `lean-ux-canvas` | 引导 Lean UX Canvas v2，澄清业务问题、假设和学习目标。 | 想快速对齐产品机会和实验方向。 |
| `opportunity-solution-tree` | 从 outcome 到机会、方案、测试构建机会方案树。 | 面对 stakeholder 需求时先做问题拆解。 |
| `pol-probe` | 定义 Proof of Life probe，低成本验证高风险假设。 | 开发前想先得到残酷真实反馈。 |
| `pol-probe-advisor` | 根据假设、风险和资源选择合适的 PoL probe。 | 不确定该用访谈、落地页、假门还是 concierge 等方法。 |
| `recommendation-canvas` | 评估 AI 产品想法的 outcome、假设、风险、定位。 | 判断 AI 方案是否值得投入。 |

## 5. 产品定义、规划与交付

| Skill | 中文说明 | 适合什么时候用 |
| --- | --- | --- |
| `prd-development` | 构建结构化 PRD，连接问题、用户、方案和成功标准。 | 把 discovery 结果变成工程可执行文档。 |
| `epic-hypothesis` | 把大型 initiative 写成可验证 epic 假设。 | 在 roadmap 或交付前定义大方向。 |
| `epic-breakdown-advisor` | 用 Humanizing Work split patterns 拆分 epic 为 stories。 | backlog 项太大，无法估算或交付。 |
| `user-story` | 用 Mike Cohn 格式和 Gherkin 验收标准写用户故事。 | 把需求变成开发可执行 story。 |
| `user-story-splitting` | 用成熟拆分模式把大 story/epic 拆小。 | story 太大，需要独立交付切片。 |
| `user-story-mapping` | 创建 story map：活动、步骤、任务、发布切片。 | 规划一个完整用户流程或 MVP。 |
| `user-story-mapping-workshop` | 交互式用户故事地图工作坊。 | 想逐步产出 backbone、tasks、release slices。 |
| `roadmap-planning` | 做战略 roadmap：优先级、epic、stakeholder、节奏。 | 把战略转成可执行发布计划。 |
| `prioritization-advisor` | 选择 RICE、ICE、价值/成本等优先级框架。 | 不确定该用哪种 prioritization 方法。 |
| `feature-investment-advisor` | 从收入影响、成本、ROI、战略角度评估 feature 投资。 | 判断某功能值不值得做。 |
| `product-strategy-session` | 端到端产品战略 session：定位、发现、路线图。 | 团队需要在执行前找到验证过的方向。 |
| `press-release` | 写 Amazon-style press release，先定义客户价值。 | 新产品/功能立项前统一愿景。 |
| `eol-message` | 写清晰、有同理心的产品/功能/套餐下线公告。 | 需要减少 EOL 带来的客户困惑。 |

## 6. 定位、市场与叙事

| Skill | 中文说明 | 适合什么时候用 |
| --- | --- | --- |
| `positioning-statement` | 创建 Geoffrey Moore 风格定位语句。 | 明确服务谁、解决什么、品类和差异化。 |
| `positioning-workshop` | 定位工作坊：目标客户、未满足需求、品类、收益、差异化。 | 产品 messaging 模糊或团队理解不一致。 |
| `storyboard` | 创建六帧 storyboard，展示用户从问题到解决方案的旅程。 | 做概念评审、对齐、demo 前叙事。 |
| `tam-sam-som-calculator` | 计算 TAM/SAM/SOM，并列出假设、方法和 caveats。 | 做市场规模、商业案例、投资论证。 |
| `pestel-analysis` | 分析政治、经济、社会、技术、环境、法律外部因素。 | 市场外部变化可能影响产品或战略。 |
| `company-research` | 公司研究简报：高管引用、产品战略、组织背景。 | 面试、竞品、合作、市场进入准备。 |

| `business-health-diagnostic` | 从增长、留存、效率、资本角度诊断 SaaS 业务健康。 | 准备 business review 或定位紧急问题。 |

## 快速选择指南

- 想做网页/应用界面：优先看 `design-taste-frontend`、`redesign-existing-projects`、`image-to-code`。
- 想做图片/品牌/视觉稿：优先看 `imagegen`、`brandkit`、`imagegen-frontend-web`、`imagegen-frontend-mobile`。
- 想做 PPT/文档/表格/PDF：优先看 `presentations:Presentations`、`documents:documents`、`spreadsheets:Spreadsheets`、`pdf`。
- 想做产品发现：优先看 `discovery-process`、`jobs-to-be-done`、`customer-journey-map`、`opportunity-solution-tree`。
- 想写 PRD / story / roadmap：优先看 `prd-development`、`user-story`、`epic-breakdown-advisor`、`roadmap-planning`。
- 想做定位和市场分析：优先看 `positioning-statement`、`positioning-workshop`、`tam-sam-som-calculator`、`company-research`。
- 想分析 SaaS 增长/财务：优先看 `business-health-diagnostic`、`saas-revenue-growth-metrics`、`finance-based-pricing-advisor`。
- 想做 AI 产品/Agent 工作流：优先看 `ai-shaped-readiness-advisor`、`context-engineering-advisor`、`recommendation-canvas`、`openai-docs`。
- 想创建/安装/管理 skills：优先看 `skill-creator`、`skill-authoring-workflow`、`skill-installer`。
