# PixelGraph 第二阶段剩余任务梳理

## 当前已完成基础

第二阶段已经不只是第一阶段 MVP 的延续，当前项目已经具备一批可用基础能力：

- 支持 ER、Class、Sequence、State、Flowchart 五类图表切换。
- Mermaid 图表具备实时渲染、错误展示、缩放、适配视图、全屏预览、SVG/PNG/Markdown 导出。
- ER 已支持 SQL 输入模式和 Mermaid ER 输入模式。
- SQL ER 已有数据库表视图、Chen ER 视图、React Flow 拖拽、缩放、自动排版、字段显示控制、颜色和节点比例设置。
- 已有本地保存能力，覆盖源码、图表类型、主题、语言、ER 输入模式、画布设置和部分 ER 显示设置。
- 已有中英文界面切换、浅色/深色主题、模板入口、导出设置入口。
- 构建已验证可通过，但存在大 chunk 警告，需要作为性能优化项跟进。

第二阶段剩余工作的重点不是继续堆新方向，而是把当前支持的图表生成质量和完整用户体验打磨到可交付状态。

## P0 必须收尾

### 1. 修复中文乱码和示例内容质量

- 目标：修复图表定义、模板名称、模板描述、示例源码和错误文案中的乱码。
- 原因：模板是用户打开产品后的第一印象，乱码会让产品显得不可用，也会影响用户理解每类图表的输入方式。
- 验收口径：
  - ER、Class、Sequence、State、Flowchart 的 tab、模板按钮、描述、默认源码均为可读中文。
  - Mermaid 示例源码中的节点名、参与者名、状态名和边标签不会出现乱码或语义断裂。
  - 中文和英文界面切换后，主要按钮、状态、设置项、错误信息均语义清楚。

### 2. 打磨所有非 ER Mermaid 图表生成质量

- 目标：分别优化 Class、Sequence、State、Flowchart 的默认模板、渲染样式和错误体验。
- 原因：当前第二阶段不能只把 SQL ER 做成亮点，其他图表也必须达到“打开即能生成一张像样图”的标准。
- 验收口径：
  - Class 图模板能表达真实模块关系，类名、字段、方法和依赖关系清晰。
  - Sequence 图模板包含真实调用链，参与者命名清楚，消息方向和返回关系可读。
  - State 图模板覆盖正常流转、失败/回退或终止路径，不只是最短示例。
  - Flowchart 图模板覆盖判断分支、异常路径和结果节点，适合放入技术文档。
  - 每类图表至少保留 2 个高质量模板，模板名称不能只是“输入某某图”。

### 3. 收口 SQL ER 编辑和模型同步

- 目标：保证在 ER 画布中编辑表、字段、关系后，SQL/model/画布三者保持一致。
- 原因：当前 ER 支持直接编辑，但部分编辑只影响临时图形或局部 model，可能造成刷新、导出或再次解析后丢失。
- 验收口径：
  - 重命名表时，同步更新引用该表的外键关系。
  - 重命名字段时，同步更新相关主键、外键和关系边。
  - 修改字段类型、注释、主键、可空状态后，源码和画布都能稳定更新。
  - 关系名称和基数如果保留编辑入口，必须能持久化；如果无法持久化，则移除或弱化相关入口。
  - 手动画线如果支持，必须回写关系；如果不支持，禁用或改成明确的只读交互。

### 4. 修复 ER 节点位置持久化作用域

- 目标：ER 节点位置按图表内容或项目状态隔离保存。
- 原因：当前节点位置是全局保存，切换 SQL、模板或图表后可能串位置，导致布局混乱。
- 验收口径：
  - 不同 SQL 模板之间的位置互不污染。
  - 用户手动拖拽后，同一份 SQL 刷新页面能恢复位置。
  - 用户点击自动排版后，可以清除当前图的手动位置并重新布局。
  - 切换数据库 ER/Chen ER 视图时，位置策略清晰，不出现节点叠在一起或飞出画布。

### 5. 完整导出链路保护

- 目标：确保 SVG、PNG、Markdown 导出在所有图表类型下都稳定可用。
- 原因：导出是 PixelGraph 的核心闭环，任何图表类型导出失败都会破坏产品定位。
- 验收口径：
  - ER、Class、Sequence、State、Flowchart 均能导出 SVG、PNG、Markdown。
  - 空源码、渲染失败、未生成 SVG 时，导出按钮有明确禁用或错误提示。
  - PNG 倍率和透明背景设置对 Mermaid 和 SQL ER 都生效。
  - 导出文件名包含图表类型和时间或稳定标识，避免反复下载覆盖。
  - 暗色主题下导出的 SVG/PNG 仍然可读。

## P1 用户体验与图表生成打磨

### 1. 编辑体验

- 目标：让用户在纯文本输入时更容易写出正确图表。
- 原因：PixelGraph 是文本驱动工具，编辑器体验直接决定生成效率。
- 验收口径：
  - 格式化按钮不只是 trim 行尾，应对 Mermaid 和 SQL 有最小可感知的格式化效果。
  - 错误状态能提示大致位置或错误类型，而不是只展示底层异常。
  - 编辑器支持常用快捷键说明或行为，例如 Tab 缩进、Ctrl/Cmd+Enter 重新渲染、Ctrl/Cmd+S 保存状态。
  - 行号和文本滚动保持对齐，长文本输入时不卡顿。
  - 占位符按当前图表类型变化，避免所有模式都提示“输入 SQL 或 Mermaid”。

### 2. 预览体验

- 目标：让预览画布在小图、大图、错误图、空图、移动端下都稳定。
- 原因：图表工具的核心感受来自预览区，布局抖动、控件拥挤或缩放失控都会放大挫败感。
- 验收口径：
  - Mermaid 预览支持拖拽平移、滚轮缩放、适配视图和重置视图，并且不会与页面滚动冲突。
  - SQL ER 预览的 React Flow 控件、MiniMap、工具条在窄屏下不遮挡主要内容。
  - 全屏预览中工具条、导出入口、退出全屏按钮可用且不重叠。
  - 渲染中、空状态、错误状态都有和最终布局一致的占位，不使用突兀的跳动。
  - 暗色主题下画布网格、节点、连线、标签和状态提示对比度合格。

### 3. 图表样式设置

- 目标：让不同图表类型的样式控制更贴近实际需求。
- 原因：当前 Mermaid 设置较通用，但 Class、Sequence、State、Flowchart 对样式参数的敏感点不同。
- 验收口径：
  - Sequence 图重点支持参与者间距、是否显示编号、是否镜像参与者。
  - Flowchart 图重点支持方向、曲线样式、节点间距、层级间距。
  - Class 图重点支持字体大小、节点间距、线条颜色和类框可读性。
  - State 图重点支持状态间距、终止节点可读性和分支连线清晰度。
  - 设置项只在相关图表类型出现，避免用户看到无效配置。

### 4. 模板系统

- 目标：让模板成为用户学习和快速产出的入口。
- 原因：模板不是装饰内容，而是降低 Mermaid/SQL 学习成本的关键。
- 验收口径：
  - 每个模板都有清楚名称、适用场景和高质量源码。
  - 模板按钮区在模板变多后仍然可扫读，支持搜索或分类可以放入后续增强。
  - 切换模板前如当前内容已修改，需要考虑确认、撤销或至少明确会替换内容。
  - 模板加载后状态提示明确，用户知道当前图来自哪个模板。

### 5. 本地保存与状态恢复

- 目标：让用户刷新页面后回到合理的工作状态。
- 原因：本地优先是产品定位之一，状态恢复必须可靠。
- 验收口径：
  - 刷新后恢复图表类型、ER 输入模式、源码、主题、语言、画布设置。
  - 设置项恢复后不会造成图表不可见，例如暗色背景配暗色文字。
  - 用户切换图表类型时，要明确是加载该类型模板，还是保留原输入。
  - 保存状态文案不要频繁闪烁，避免影响编辑专注。

### 6. UI 和视觉 polish

- 目标：让界面从“功能堆叠”变成稳定的开发者工作台。
- 原因：第二阶段需要提高可用性和信任感，尤其是长时间编辑体验。
- 验收口径：
  - 顶栏、设置抽屉、模板区、编辑器、预览区、导出区都有清晰层级。
  - 按钮文案和按钮尺寸适配中英文，不出现文字挤压或换行破坏布局。
  - 图标按钮需要有可访问名称，文本按钮语义明确。
  - 移动端按工具栏、编辑器、预览、状态、导出的顺序堆叠，不互相遮挡。
  - 不引入大面积渐变、装饰性图形或营销页式布局，保持工具属性。

### 7. 工程质量和性能

- 目标：让第二阶段具备稳定迭代基础。
- 原因：当前依赖 Mermaid 和 React Flow，包体和渲染成本都需要可控。
- 验收口径：
  - `npm.cmd run build` 稳定通过。
  - 处理 Vite 大 chunk 警告，优先考虑 Mermaid 动态加载、图表类型懒加载或 manualChunks。
  - 为 SQL 解析、SQL model 转换、导出保护添加基础测试。
  - 为五类图表准备人工验收样例，覆盖成功、错误、空输入和大图。

## P2 可延后增强

以下能力有价值，但不应阻塞第二阶段交付：

- 多图项目管理、项目 JSON 导入/导出、文件夹、标签、搜索。
- 用户自定义模板库和模板收藏。
- 更完整的 CodeMirror 或 Monaco 编辑器集成，包括语法高亮、自动补全、错误定位。
- 更多 SQL 方言支持，例如 PostgreSQL、SQL Server、Oracle 的完整建表语法。
- 图表历史版本、撤销/重做栈、差异对比。
- 大图 Web Worker 渲染和更细粒度性能监控。
- PDF、JPEG、draw.io XML 等更多导出格式。

AI 生成、桌面端、VS Code 插件、云同步、多人协作属于后续大阶段方向，不纳入第二阶段必做范围。

## 验收清单

### 图表类型验收

- ER：SQL 输入、Mermaid ER 输入、数据库 ER 视图、Chen ER 视图、拖拽、自动排版、字段显示、导出。
- Class：默认模板、编辑后实时渲染、样式设置、错误提示、导出。
- Sequence：参与者、消息、返回、编号设置、镜像设置、导出。
- State：初始状态、终止状态、分支状态、错误提示、导出。
- Flowchart：方向、判断节点、异常路径、线条样式、导出。

### 用户体验验收

- 首次打开页面能看到可读、专业、无乱码的默认示例。
- 切换任意图表类型后，编辑器和预览区都能进入可用状态。
- 输入错误时，用户能理解下一步该改什么。
- 刷新页面后能恢复最近工作内容。
- 浅色和深色主题都可读。
- 桌面端和移动端都不出现控件重叠。
- 全屏预览可进入、可退出、可继续查看图表。
- 导出失败时有明确反馈，不静默失败。

### 工程验收

- 构建通过。
- 主要新增逻辑有测试或人工验收样例。
- 浏览器控制台无明显运行时错误。
- `git status --short` 中只包含本次预期变更。

## 建议执行顺序

1. 先修复中文乱码、模板内容和默认示例，立刻提升第一印象。
2. 分别打磨 Class、Sequence、State、Flowchart 的模板、样式和错误体验，补齐非 ER 图表质量。
3. 收口 SQL ER 的编辑、关系、节点位置和导出一致性，避免专业能力半成品。
4. 统一编辑器、预览区、导出区、本地保存的用户体验细节。
5. 做桌面端、移动端、浅色/深色、全屏、刷新恢复的完整人工验收。
6. 最后处理构建包体、基础测试和验收样例，把第二阶段固定成稳定版本。
