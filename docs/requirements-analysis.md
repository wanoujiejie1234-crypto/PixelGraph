# PixelGraph 需求分析

## 1. 项目背景

在日常开发、系统设计、技术文档编写和方案评审中，开发者经常需要绘制各种软件工程图，例如：

- E-R 图
- UML 类图 / 实体类图
- 状态图
- 顺序图
- 流程图
- 架构图
- 用例图
- 数据流图

现有在线画图工具通常存在以下问题：

- 免费版本限制较多
- 高级导出能力需要付费
- 在线保存存在隐私和数据安全顾虑
- 手动拖拽画图效率较低
- 图形调整、对齐和排版比较耗时
- 不方便和 Markdown、README、接口文档、技术方案结合

因此，本项目希望实现一个面向开发者的免费图表生成工具，让用户通过文本、结构化描述或后续的代码解析能力，快速生成常见软件设计图。

## 2. 项目定位

PixelGraph 定位为：

> 一个免费的、开发者优先的本地图表生成器，用文本快速生成软件设计图。

核心价值：

- 免费可用
- 本地优先
- 开发者友好
- 输入简单
- 实时预览
- 导出方便
- 适合技术文档场景

## 3. 目标用户

主要目标用户包括：

- 后端开发者
- 全栈开发者
- 架构师
- 编程学习者
- 计算机相关专业学生
- 技术文档编写者
- 中小团队开发者

用户特点：

- 熟悉代码、Markdown、JSON、YAML 等文本格式
- 更偏好用文本描述结构，而不是反复拖拽图形
- 需要快速生成能放入文档、README、Wiki 或 PPT 的图
- 希望工具轻量、免费、无需登录
- 对本地运行和数据隐私有一定诉求

## 4. 核心使用场景

### 4.1 数据库设计

用户需要快速根据实体、字段、主键、外键和表关系生成 E-R 图。

典型场景：

- 新项目数据库设计
- 数据表关系梳理
- 老系统数据库文档补充
- 面试、课程设计或毕业设计文档

### 4.2 面向对象建模

用户需要生成类图或实体类图，用于描述系统对象、字段、方法和继承关系。

典型场景：

- 领域模型设计
- 实体类结构说明
- 类之间关系梳理
- 技术方案评审

### 4.3 接口调用和业务链路说明

用户需要通过顺序图描述用户、前端、后端服务、数据库、第三方系统之间的调用流程。

典型场景：

- 登录流程
- 下单流程
- 支付流程
- 审批流程
- 微服务调用链路

### 4.4 状态流转设计

用户需要描述订单、任务、审批、工单等对象在不同状态之间的迁移关系。

典型场景：

- 订单状态机
- 审批流状态
- 任务生命周期
- 支付状态流转

### 4.5 业务流程说明

用户需要绘制流程图，用于表达业务规则、算法逻辑和系统处理步骤。

典型场景：

- 需求说明
- 接口逻辑说明
- 异常分支梳理
- 业务流程文档

## 5. 功能需求

### 5.1 图表类型支持

第一阶段建议支持以下图表：

- E-R 图
- UML 类图 / 实体类图
- 顺序图
- 状态图
- 流程图

后续可扩展：

- 架构图
- 用例图
- 甘特图
- C4 模型图
- 数据流图
- 思维导图

### 5.2 文本编辑能力

系统应提供图表描述文本的编辑能力。

基础能力：

- 文本输入
- 多行编辑
- 基础语法提示
- 错误提示
- 快捷键支持
- 自动保存草稿

增强能力：

- 语法高亮
- 自动补全
- 格式化
- 折叠代码块
- 多标签编辑

### 5.3 实时预览能力

用户输入图表描述后，系统应实时生成图表预览。

需求包括：

- 输入变更后自动刷新预览
- 渲染失败时显示错误信息
- 支持缩放
- 支持拖动画布
- 支持适配屏幕
- 支持重新渲染

### 5.4 模板能力

系统应提供常用图表模板，降低用户上手成本。

第一阶段模板：

- 用户订单 E-R 图
- 登录顺序图
- 订单状态图
- 基础类图
- 简单业务流程图

模板应支持：

- 一键插入编辑器
- 按图表类型分类
- 用户可基于模板修改

### 5.5 导出能力

用户应能将生成的图表导出到常见格式。

第一阶段支持：

- SVG
- PNG
- Mermaid 源码
- Markdown 代码块

后续可支持：

- PDF
- JPEG
- 项目 JSON
- draw.io XML

### 5.6 本地项目管理

第一阶段可使用浏览器本地存储实现轻量项目管理。

基础能力：

- 新建图表
- 保存图表
- 重命名图表
- 删除图表
- 最近编辑记录
- 导入 / 导出项目文件

后续可扩展：

- 文件夹管理
- 标签分类
- 搜索
- 版本历史
- 云端同步

### 5.7 主题和显示设置

系统应支持基础显示设置。

需求包括：

- 浅色模式
- 深色模式
- 图表主题切换
- 编辑器字体大小调整
- 预览区域缩放

## 6. 非功能需求

### 6.1 易用性

- 打开即可使用
- 无需注册登录
- 默认提供示例内容
- 图表类型切换清晰
- 错误信息尽量可读

### 6.2 性能

- 输入后预览刷新应尽量流畅
- 常见中小型图表应在 1 秒内完成渲染
- 大图表需要避免频繁阻塞 UI
- 可对输入进行防抖处理

### 6.3 隐私

- 第一阶段优先本地运行
- 图表内容默认不上传服务器
- 本地存储数据应由用户主动清除或导出

### 6.4 可维护性

- 图表类型渲染逻辑应模块化
- 模板数据应结构化管理
- 导出逻辑应独立封装
- 后续可接入 SQL 解析、代码解析和 AI 生成能力

## 7. MVP 范围

第一版建议做成单页 Web 应用，核心目标是跑通“输入文本 -> 实时生成图 -> 导出”的闭环。

MVP 包含：

- 图表类型选择
- Mermaid 语法输入
- E-R 图预览
- 类图预览
- 顺序图预览
- 状态图预览
- 流程图预览
- 示例模板
- 实时预览
- 渲染错误提示
- 复制源码
- 导出 SVG
- 导出 PNG
- 导出 Markdown
- 本地保存最近编辑内容
- 浅色 / 深色模式

MVP 暂不包含：

- 账号系统
- 云端同步
- 多人协作
- 拖拽编辑器
- AI 自动生成
- SQL 自动解析
- 代码自动解析
- 权限系统
- 复杂项目空间

## 8. 产品形态建议

第一阶段建议采用 Web 单页应用。

推荐布局：

- 顶部：应用名称、图表类型选择、主题切换、导出入口
- 左侧：文本编辑器
- 右侧：图表实时预览
- 底部或侧边：错误信息、模板列表、复制按钮

核心交互流程：

1. 用户选择图表类型
2. 系统加载对应示例模板
3. 用户编辑文本
4. 系统实时渲染图表
5. 用户导出 SVG、PNG 或 Markdown

## 9. 技术方案建议

### 9.1 前端技术栈

推荐：

- React
- TypeScript
- Vite
- Mermaid.js
- CodeMirror 或 Monaco Editor
- CSS Modules / Tailwind CSS / 普通 CSS

### 9.2 存储方案

MVP：

- LocalStorage：保存当前草稿、主题、最近使用图表类型
- IndexedDB：后续用于保存多个图表项目

后续：

- 本地文件导入导出
- SQLite
- 云端同步

### 9.3 图表渲染方案

MVP 可基于 Mermaid.js 实现主要图表渲染。

优势：

- 支持多种开发常用图表
- 文本语法成熟
- 社区使用广泛
- 方便导出 SVG
- 适合 Markdown 文档场景

### 9.4 导出方案

- SVG：直接获取 Mermaid 渲染后的 SVG
- PNG：将 SVG 绘制到 Canvas 后导出
- Markdown：生成 Mermaid fenced code block

## 10. 后续扩展方向

### 10.1 SQL 转 E-R 图

用户粘贴建表 SQL，系统自动识别：

- 表名
- 字段
- 类型
- 主键
- 外键
- 表关系

并生成 E-R 图。

### 10.2 代码转类图

支持从 Java、TypeScript、Python 等代码中解析类结构，生成类图。

可识别：

- 类名
- 属性
- 方法
- 继承关系
- 接口实现
- 依赖关系

### 10.3 AI 辅助生成

用户输入自然语言描述，系统生成 Mermaid 图表。

示例：

> 帮我生成一个用户下单支付的顺序图。

### 10.4 VS Code 插件

将图表生成能力集成到 VS Code：

- Markdown 内预览
- Mermaid 快速生成
- 项目内图表管理
- 从代码生成图表

### 10.5 桌面应用

后续可基于 Tauri 或 Electron 封装桌面版本。

能力包括：

- 离线使用
- 本地文件管理
- 系统菜单
- 快捷键
- 更强的隐私保障

## 11. 风险与挑战

### 11.1 Mermaid 语法学习成本

虽然 Mermaid 适合开发者，但对新用户仍有学习门槛。

应对方式：

- 提供模板
- 提供示例
- 提供错误提示
- 后续支持表单化输入或自然语言生成

### 11.2 图表复杂后的布局问题

复杂图表可能出现节点拥挤、连线混乱、预览过大等问题。

应对方式：

- 支持缩放和拖拽
- 支持布局方向设置
- 提供大图导出
- 提供拆分图表建议

### 11.3 导出图片兼容性

SVG 转 PNG 可能遇到字体、样式、跨浏览器兼容问题。

应对方式：

- 优先保证 SVG 导出稳定
- PNG 导出增加兼容处理
- 提供导出前预览

## 12. 第一阶段验收标准

第一阶段完成后，应满足：

- 用户可以打开应用并看到默认示例图
- 用户可以切换 E-R 图、类图、顺序图、状态图和流程图
- 用户可以编辑文本并实时看到图表变化
- 输入错误时能看到明确错误提示
- 用户可以复制 Mermaid 源码
- 用户可以导出 SVG
- 用户可以导出 PNG
- 用户可以导出 Markdown
- 用户刷新页面后能恢复最近编辑内容
- 应用无需登录即可使用
