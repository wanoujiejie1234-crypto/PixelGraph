## 1. 暗色模式（colorMode 硬编码）

### 缺陷

全部 4 个 Canvas 组件均硬编码 `colorMode="light"`：

```typescript
// ActivityCanvas.tsx:253
<ReactFlow colorMode="light" ... />

// UseCaseCanvas.tsx:522
<ReactFlow colorMode="light" ... />

// SqlErCanvas.tsx:561
<ReactFlow colorMode="light" ... />

// StructureCanvas.tsx:364
<ReactFlow colorMode="light" ... />
```

当应用切换为暗色主题时，ReactFlow 画布的背景、网格、控件、minimap 仍然保持亮色模式，导致视觉割裂。

### 方案

从 props 或 context 获取当前主题，动态传入 `colorMode`：

```typescript
// 在每个 Canvas 组件中
interface Props {
  // ... existing props
  themeMode?: 'light' | 'dark';  // 新增
}

// 渲染时
<ReactFlow colorMode={themeMode === 'dark' ? 'dark' : 'light'} ... />
```

在 `App.tsx` 中传入 `themeMode`（已有 `theme` 状态）：

```typescript
<ActivityCanvas themeMode={theme} ... />
```

## 2. Structure 容器节点视觉无区分

### 缺陷

`structureFlowNodes.tsx:77-114` 中 `StructureContainerNode` 对 `cloud`、`node`、`device`、`database` 四种节点全部使用同一个 CSS 类 `structure-deployment-node`，唯一的区别是顶部 `<<stereotype>>` 文字标签：

```typescript
const className = props.data.kind === 'package'
  ? 'structure-package-node'
  : props.data.kind === 'execution'
    ? 'structure-execution-node'
    : ['cloud', 'node', 'device', 'database'].includes(props.data.kind)
      ? 'structure-deployment-node'  // ← 四种不同元素共用同一个样式
      : 'structure-group-node';
```

UML 标准中，cloud（云边界）、node（节点/立方体）、device（设备）、database（数据库柱体）应有完全不同的视觉形状。

### 方案

为每种部署节点创建独立的渲染分支：

```typescript
// structureFlowNodes.tsx
export function StructureCloudNode(props: NodeProps<StructureGraphNode>) {
  return (
    <div className="structure-cloud-node">
      <div className="structure-cloud-shape" aria-hidden="true">
        <svg viewBox="0 0 120 80">
          <ellipse cx="60" cy="40" rx="55" ry="30" />
          <path d="M5 40 Q5 10 35 10 Q60 10 70 25 Q90 15 105 25 Q115 30 115 50 Q115 70 90 70 L20 70 Q5 70 5 40Z" />
        </svg>
      </div>
      <EditableLabel ... />
    </div>
  );
}

export function StructureDatabaseNode(props: NodeProps<StructureGraphNode>) {
  return (
    <div className="structure-database-node">
      {/* 柱体形状：椭圆顶 + 矩形体 + 椭圆底 */}
      <svg viewBox="0 0 100 120">
        <ellipse cx="50" cy="15" rx="40" ry="12" />
        <path d="M10 15 L10 90 Q10 105 50 105 Q90 105 90 90 L90 15" />
        <ellipse cx="50" cy="90" rx="40" ry="12" />
      </svg>
      <EditableLabel ... />
    </div>
  );
}

export function StructureDeviceNode(props: NodeProps<StructureGraphNode>) {
  return (
    <div className="structure-device-node">
      {/* 设备：立方体形状 */}
      <svg viewBox="0 0 100 80">
        <rect x="5" y="15" width="90" height="60" rx="4" />
        <rect x="30" y="60" width="40" height="4" rx="2" />
        <rect x="15" y="5" width="70" height="12" rx="3" />
      </svg>
      <EditableLabel ... />
    </div>
  );
}
```

在 `StructureCanvas.tsx` 中注册新节点类型：

```typescript
const nodeTypes = {
  'structure-cloud': StructureCloudNode,
  'structure-database': StructureDatabaseNode,
  'structure-device': StructureDeviceNode,
  // ...
};
```

## 3. Chen ER 前端渲染已就绪，但模型未接入

### 缺陷

`ErFlowNodes.tsx` 前端渲染组件**已经完整实现**了所有 Chen ER 变体：

| 元素 | CSS 类存在 | 渲染组件存在 | 模型数据未填充 |
|------|-----------|------------|--------------|
| 弱实体 `entityKind='weak'` | `.chen-entity-frame-inner` 双线 | `ChenEntityGeometry` 中的 `weak` 分支 | ✅ |
| 联合实体 `entityKind='associative'` | `.chen-entity-frame-associative` 内三角 | `ChenEntityGeometry` 中的 `associative` 分支 | ✅ |
| 标识关系 `relationshipKind='identifying'` | `.chen-relationship-diamond-inner` 双线 | `ChenRelationshipNode` 中的 `identifying` 分支 | ✅ |
| 派生属性 `attributeKind='derived'` | `.chen-flow-attribute-derived` 虚线 | `ChenAttributeGeometry` 中的 `derived` 分支 | ✅ |
| 多值属性 `attributeKind='multivalued'` | `.chen-attribute-oval-inner` 双线 | `ChenAttributeGeometry` 中的 `multivalued` 分支 | ✅ |

但 `erGraphModel.ts:293-319` 构建 Chen 节点时，使用的是 `sqlErModel.ts` 的 `SqlTable`/`SqlColumn`/`SqlRelationship`，而这些类型中缺少 `entityKind`、`attributeKind`、`isWeakKey` 等字段。

### 方案

在 `sqlErModel.ts` 中添加字段：

```typescript
// sqlErModel.ts
export interface SqlTable {
  // ... existing
  entityKind?: 'strong' | 'weak' | 'associative';  // 新增
}

export interface SqlRelationship {
  // ... existing
  relationshipKind?: 'nonIdentifying' | 'identifying';  // 新增
}

export interface SqlColumn {
  // ... existing
  attributeKind?: 'normal' | 'derived' | 'multivalued';  // 新增
  isWeakKey?: boolean;  // 新增
}
```

在 SQL 解析器（`sqlErModel.ts` 的 `parseSql`）中检测 `WEAK` 关键字：

```sql
CREATE TABLE order_item WEAK (
  order_id BIGINT,
  PRIMARY KEY (order_id, product_id)
);
```

或在表格节点右键菜单中添加切换选项。

## 4. 活动图 Decision/Merge 无视觉区分

### 缺陷

`activityFlowNodes.tsx:93-103` 中 Decision 和 Merge 使用完全相同的组件 `ActivityDecisionNode`，仅通过 `is-merge` CSS 类做微小差异：

```typescript
export function ActivityDecisionNode(props: NodeProps<ActivityGraphNode>) {
  const isMerge = props.data.kind === 'merge';
  return (
    <div className={`... ${isMerge ? 'is-merge' : ''}`}>
      ...
    </div>
  );
}
```

UML 标准中 Decision（菱形，带条件标签）和 Merge（菱形，无内容）应有更明显的视觉区分。当前 Merge 节点显示为小菱形，但和常规 Decision 差别不够明显。

### 方案

为 Merge 节点添加不同的尺寸和样式：

```css
/* styles.css */
.activity-decision-node.is-merge .activity-decision-shape {
  width: 22px;
  height: 22px;
  border-style: dashed;  /* Merge 使用虚线菱形 */
}
```

或者在 Merge 内部添加连接线标记（三条汇聚线的小图标）。

## 8. UseCase Note 与 Constraint 视觉区分不足

### 缺陷

`useCaseFlowNodes.tsx:110-122` 中 Note 和 Constraint 共用同一节点 `UseCaseNoteNode`，Constraint 仅在底部显示 `{constraint}` 文字：

```typescript
{note?.kind === 'constraint' ? <p className="structure-note-target">{'{constraint}'}</p> : null}
```

对比 UML 标准：Constraint 应以 `{text}` 大括号形式展示在被约束元素附近，而非独立节点。

### 方案

将 Constraint 改为在关联目标节点下方以内联方式显示：

```typescript
// UseCaseNoteNode 中
if (note?.kind === 'constraint') {
  return (
    <div className="usecase-constraint-inline">
      <span className="usecase-constraint-brace">{`{ ${note.text} }`}</span>
    </div>
  );
}
```

```css
.usecase-constraint-inline {
  font-family: var(--mono);
  color: var(--steel);
  font-size: 0.85em;
  padding: 2px 6px;
}
```

Note 则保持独立节点带折叠角样式。

## 9. 跨 Canvas 的 SVG Marker 定义冗余

### 缺陷

每个 Canvas 组件的 SVG `<defs>` 中都独立定义了箭头标记，存在大量重复：

| Marker ID | 定义在 | 也定义在 |
|-----------|--------|---------|
| `pg-activity-arrow` | ActivityCanvas | — |
| `pg-activity-open-arrow` | ActivityCanvas | — |
| `pg-usecase-open-arrow` | UseCaseCanvas | — |
| `pg-usecase-hollow-triangle` | UseCaseCanvas | — |
| `pg-structure-open-arrow` | StructureCanvas | — |
| `pg-structure-solid-arrow` | StructureCanvas | — |
| `pg-structure-hollow-triangle` | StructureCanvas | — |

泛化箭头标记在 UseCaseCanvas 和 StructureCanvas 中各自定义，如果样式不同会造成视觉不一致。

### 方案

统一迁移到全局 SVG defs，在 `App.tsx` 或独立 `markers.tsx` 中定义：

```typescript
// 新增 src/features/renderer/markers.tsx
export function DiagramMarkers({ color = '#171817' }: { color?: string }) {
  return (
    <svg height="0" width="0" aria-hidden="true" focusable="false">
      <defs>
        <marker id="pg-arrow" viewBox="0 0 10 10" refX="9" refY="5" markerWidth="10" markerHeight="10" orient="auto">
          <path d="M1 1 9 5 1 9" fill="none" stroke={color} strokeLinecap="round" strokeLinejoin="round" strokeWidth="1.5" />
        </marker>
        <marker id="pg-hollow-triangle" viewBox="0 0 14 12" refX="12" refY="6" markerWidth="14" markerHeight="12" orient="auto">
          <path d="M1.5 1.5 12 6 1.5 10.5Z" fill="var(--er-fill, white)" stroke={color} strokeLinejoin="round" strokeWidth="1.5" />
        </marker>
        <marker id="pg-solid-diamond" viewBox="0 0 12 12" refX="10" refY="6" markerWidth="12" markerHeight="12" orient="auto">
          <path d="M2 6 6 2 10 6 6 10Z" fill={color} stroke={color} strokeWidth="1" />
        </marker>
        <marker id="pg-hollow-diamond" viewBox="0 0 12 12" refX="10" refY="6" markerWidth="12" markerHeight="12" orient="auto">
          <path d="M2 6 6 2 10 6 6 10Z" fill="var(--er-fill, white)" stroke={color} strokeWidth="1.5" />
        </marker>
      </defs>
    </svg>
  );
}
```

然后在 `App.tsx` 中渲染一次全局标记，各 Canvas 组件引用的 marker URL 统一定位到全局 SVG 中的 ID。

## 10. 空状态/无内容引导缺失

### 缺陷

当 DSL 内容为空或解析失败时，所有 Canvas 显示一个错误状态 div，但没有"空白画布"的引导提示。用户打开一个空模板时，看到的是空白画布加工具栏，不清楚如何开始。

### 方案

在每个 Canvas 中添加空状态引导：

```typescript
// 在 InnerActivityCanvas / InnerUseCaseCanvas 等中
if (graph.nodes.length === 0 && !validation.hasFatalError) {
  return (
    <div className="er-flow-shell">
      <div className="er-empty-state">
        <strong>开始绘制</strong>
        <p>在左侧编辑器中输入 {diagramType} DSL 源码，画布将自动生成可拖拽的图形。</p>
        <button onClick={loadDefaultTemplate}>加载示例模板</button>
      </div>
    </div>
  );
}
```

## 11. Activity 节点位置不持久化

### 缺陷

`ActivityCanvas.tsx` 只持久化节点 positions（`writeStoredUmlNodePositions`），但未持久化节点 sizes。而 `UseCaseCanvas` 和 `StructureCanvas` 同时持久化了 positions 和 sizes。

这意味着用户调整活动图节点大小后刷新页面，size 会丢失。

### 方案

参照 `UseCaseCanvas.tsx` 的模式，在 `ActivityCanvas.tsx` 中添加：

```typescript
const [nodeSizes, setNodeSizes] = useState(() => readStoredUmlNodeSizes(contentScope));

useEffect(() => {
  writeStoredUmlNodeSizes(contentScope, nodeSizes);
}, [contentScope, nodeSizes]);
```

在 `buildActivityGraph` 的结果中，应用持久化的 sizes 覆盖默认尺寸。

## 12. UseCase 泛化连线使用直线路径

### 缺陷

`useCaseFlowEdges.tsx:155-172` 中 `UseCaseGeneralizationEdge` 使用 `getStraightPath` 渲染泛化连线，而 UML 标准中泛化连线和普通关联连线应采用相同的路径风格（smooth step）：

```typescript
export function UseCaseGeneralizationEdge(props: EdgeProps<UseCaseGraphEdge>) {
  const [path] = getStraightPath(props);  // ← 直线路径
```

### 方案

改为与关联连线相同的路径算法：

```typescript
const [path] = getUseCasePath(props);  // ← 复用 useCasePath，支持 smooth/bezier/straight
```

## 13. UseCase 关联无箭头且不可标注方向

### 缺陷

`UseCaseAssociationEdge` 的 `markerEnd` 根据 `associationArrow` 设置，默认值为 `'none'`（无箭头）：

```typescript
// useCaseModel.ts:137
associationArrow: 'none',
```

但 UML 标准中，关联可以在任一端加箭头表示导航方向。当前用户无法在 DSL 中指定关联方向。

### 方案

在 DSL 中新增方向语法：

```
associations
  顾客 -> 浏览商品    [→]  左到右
  顾客 <- 浏览商品    [←]  右到左
  顾客 <-> 浏览商品   [↔]  双向
```

在 `parseUseCaseModel` 中解析方向标记，设置 `associationArrow` 为 `'open'`，并在对应的边上添加 `markerStart` 或 `markerEnd`。

### 缺陷

Class、Sequence、State、Flowchart 四类图仅通过 Mermaid 渲染为静态 SVG，完全没有 React Flow Canvas。

## 15. 其他前端视觉问题

### 15.1 ER 表格节点 PK/FK 徽标重叠

当列名过长且同时有 PK/FK 徽标和数据类型标签时，`er-column-row` 的内容可能溢出。

**方案**：使用 CSS grid 确保徽标、列名、类型三列自适应，溢出时省略列名。

### 15.2 导出 SVG 中的 marker 引用

当前 SVG 导出（`activitySvgExport.ts`、`useCaseSvgExport.ts`、`structureSvgExport.ts`、`erSvgExport.ts`）需要将 React Flow 渲染的 Canvas 内嵌 marker 一并导出。如果 marker 定义在 Canvas 外部的全局 SVG 中，导出时需要包含 marker defs。

**方案**：导出函数检测并收集引用的 marker ID，将对应的 marker 定义注入到导出的 SVG 中。

### 15.3 Canvas 切换时布局状态未清理

在 App.tsx 中切换图类型时，旧 Canvas 的节点位置、大小、标签等状态未清空，新 Canvas 可能会读到残留的 localStorage 数据。

**方案**：在 Canvas 卸载时清除 scope 相关的存储数据，或在 App.tsx 中监听 `diagramType` 变化重置所有状态。
