## 1. 暗色模式支持缺失

### 缺陷

全部 4 个 Canvas 组件均硬编码 `colorMode="light"`：

```
ActivityCanvas.tsx:253 / UseCaseCanvas.tsx:522 / SqlErCanvas.tsx:561 / StructureCanvas.tsx:364
```

```typescript
<ReactFlow colorMode="light" ... />
```

当应用切换为暗色主题时，画布背景、网格、控件、minimap 保持亮色，视觉割裂。

### 解决方案

**前端层改动**：

1. 每个 Canvas 组件的 Props 中新增 `themeMode`：
```typescript
// ActivityCanvas.tsx、UseCaseCanvas.tsx、SqlErCanvas.tsx、StructureCanvas.tsx
interface Props {
  // ... existing
  themeMode?: 'light' | 'dark';
}
```

2. ReactFlow 传入动态 colorMode：
```typescript
<ReactFlow colorMode={themeMode === 'dark' ? 'dark' : 'light'} ... />
```

3. 在 `App.tsx` 传入 theme（已有 `theme` 状态）：
```tsx
<ActivityCanvas themeMode={theme} ... />
<UseCaseCanvas themeMode={theme} ... />
<SqlErCanvas themeMode={theme} ... />
<StructureCanvas themeMode={theme} ... />
```

## 3. Chen ER 渲染已就绪但模型未接入

### 缺陷

**前端 `ErFlowNodes.tsx` 已经完整实现**了所有 Chen ER 变体的渲染：

```typescript
// ChenEntityGeometry 已支持: entityKind = 'strong' | 'weak' | 'associative'
// ChenAttributeGeometry 已支持: attributeKind = 'normal' | 'derived' | 'multivalued'
// ChenRelationshipNode 已支持: relationshipKind = 'nonIdentifying' | 'identifying'
// CSS 中已有: .chen-entity-frame-inner, .chen-relationship-diamond-inner,
//            .chen-flow-attribute-derived, .chen-attribute-oval-inner
```

但 `sqlErModel.ts` 的 `SqlTable`/`SqlColumn`/`SqlRelationship` 类型中**缺少这些字段**，解析器不会填充，所以前端从未渲染出双线框/虚线。

| 元素 | 前端组件就绪 | CSS 就绪 | 模型类型缺字段 |
|------|------------|---------|--------------|
| 弱实体（双线矩形） | `ChenEntityGeometry weak` 分支 | `.chen-entity-frame-inner` | `SqlTable.entityKind` |
| 联合实体（内三角） | `ChenEntityGeometry associative` 分支 | `.chen-entity-frame-associative` | `SqlTable.entityKind` |
| 标识关系（双线菱形） | `ChenRelationshipNode` 内 `identifying` 分支 | `.chen-relationship-diamond-inner` | `SqlRelationship.relationshipKind` |
| 派生属性（虚线椭圆） | `ChenAttributeGeometry derived` 分支 | `.chen-flow-attribute-derived` | `SqlColumn.attributeKind` |
| 多值属性（双线椭圆） | `ChenAttributeGeometry multivalued` 分支 | `.chen-attribute-oval-inner` | `SqlColumn.attributeKind` |
| 弱键（下划线） | `isWeakKey` 标记 | `.chen-flow-attribute.is-weak-key` | `SqlColumn.isWeakKey` |
| ER 泛化/继承 | ❌ 不存在 | ❌ 不存在 | ❌ 不存在 |

### 解决方案

**后端层改动** — 在 `sqlErModel.ts` 补全字段：

```typescript

}

}

export interface SqlColumn {
  // ... 现有字段
  attributeKind?: 'normal' | 'derived' | 'multivalued';  // 新增
  isWeakKey?: boolean;  // 新增
}

// 新增泛化关系模型

  superEntity: string;
  subEntity: string;

```

SQL 解析器中检测 `WEAK` 关键字填充 `entityKind`：


## 7. 活动图缺少异常处理器

### 缺陷

UML 活动图支持异常处理（Exception Handler），当前模型和前端均不存在。

### 解决方案

**后端层改动** — 模型和 DSL：

```typescript
// activityModel.ts
export type ActivityNodeKind = 'lane' | 'start' | 'end' | 'flowFinal' | 'action'
  | 'object' | 'decision' | 'merge' | 'fork' | 'join' | 'note'
  | 'handlerBody';  // 新增

// ActivityStatement 新增 try 语句
| { id: string; type: 'try'; body: ActivityStatement[]; handlers: Array<{ variable: string; body: ActivityStatement[] }> }
```

DSL 语法：

```
try
  :危险操作;
catch (Exception e)
  :记录错误;
  :发送告警;
end try
```

在 `parseBlock` 中识别 `try` / `catch` / `end try`：

```typescript
if (/^try$/iu.test(line)) {
  advance(state);
  const body = parseBlock(state, ['catch', 'end try']);
  const handlers: Array<{ variable: string; body: ActivityStatement[] }> = [];
  while (state.index < state.lines.length) {
    const nextLine = currentLine(state)?.trim();
    if (/^end try$/iu.test(nextLine ?? '')) break;
    const catchMatch = nextLine?.match(/^catch\s*\((.+)\)$/iu);
    if (!catchMatch) break;
    advance(state);
    const handlerBody = parseBlock(state, ['catch', 'end try']);
    handlers.push({ variable: catchMatch[1].trim(), body: handlerBody.statements });
    if (handlerBody.stopToken === 'end try') break;
  }
  advance(state);
  statements.push({ id: nextId(state, 'try'), type: 'try', body, handlers });
  continue;
}
```

**前端层改动**：

```typescript
// activityFlowNodes.tsx
export function ActivityHandlerNode(props: NodeProps<ActivityGraphNode>) {

      <Handles />

      <EditableLabel className="activity-handler-label" node={props} />
    </div>
  );
}
```

## 17. 用例图泛化未处理关联继承

### 缺陷

子参与者理论上应继承父参与者的用例关联，但当前仅记录 generalization 关系，未在连线时自动加入继承的关联。

### 解决方案

**前端层改动** — 在 `buildUseCaseGraph` 中添加：

```typescript
// useCaseModel.ts buildUseCaseGraph 中
const inheritedAssociations: Association[] = [];
model.generalizations.forEach(gen => {
  const isActorPair = model.actors.some(a => a.name === gen.from)
    && model.actors.some(a => a.name === gen.to);
  if (!isActorPair) return;

  // 查找父参与者的所有关联
  const parentAssociations = model.associations.filter(a => a.actor === gen.to);
  parentAssociations.forEach(pa => {
    // 如果子参与者没有同名关联，则隐式添加
    const exists = model.associations.some(
      a => a.actor === gen.from && a.useCase === pa.useCase
    );
    if (!exists) {
      inheritedAssociations.push({ actor: gen.from, useCase: pa.useCase });
    }
  });
});

// 将继承的关联加入 graph
const allAssociations = [...model.associations, ...inheritedAssociations];
```
